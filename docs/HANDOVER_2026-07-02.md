### App Env For Demo Preview OCR

For a demo that must feel alive quickly:

```text

```

This means "preview OCR the first three pages". The current code does not
automatically run a later full-document enrichment when `MARKER_PAGE_RANGE`
returns successfully. Do not set `MARKER_PAGE_RANGE` in production unless
partial extraction is acceptable or a full-document async path has been added.

Update 2026-07-07: partial extraction is no longer silent. Migration 033 adds
`app.notes.extraction_coverage` (jsonb: source, page_range, partial,
extracted_at), written by both import pipelines, so page-limited notes are
queryable (`extraction_coverage->>'partial' = 'true'`) and re-enrichable. The
runpod server now echoes the applied `page_range` in its response, and the
homelab marker server rejects `page_range` requests with a 400 instead of
silently returning the full document. The automatic enrichment pass itself is
still not built.
Do not promise full-course, full-PDF OCR in 15 minutes from the numbers captured
so far.

RunPod's HTTP proxy is convenient for demo integration, but current RunPod docs
state that the proxy path goes through Cloudflare and has a 100-second maximum
connection time before a `524` timeout. For large/full-document tests, prefer
TCP/direct access, an SSH tunnel, or an async job API rather than one long HTTP
request through the proxy.

Main findings:

- Our FastAPI service path does not reuse upstream CLI tuning. The CLI sets
  OpenMP/BLAS thread caps, torch thread counts, and Surya batch sizes. Our
  wrapper currently sets `pdftext_workers` and relies on uvicorn process
  workers.
- Each uvicorn worker loads its own model copy. More workers can increase
  throughput, but they also burn VRAM and create CPU/thread contention.
- Marker processes whole documents per request. It does not batch pages across
  multiple PDFs inside one model call.
- `page_range` is real and useful. It is zero-based, so `0-2` means user-facing
  pages 1-3.
- Full PDF conversion does unavoidable PDF work before/around inference.
  `DocumentBuilder` renders every selected page twice, at low and high
  resolution, and `PdfProvider` also runs pdftext extraction unless forced
  otherwise.

Reconciliation note 2026-07-03: three different per-worker VRAM numbers are in
play and they are not the same thing. Measured idle model footprint on the B200
was about 3.95 GiB/worker. The planning assumption below is 7 GB/worker
including runtime headroom (this matches upstream `marker/utils/batch.py`'s
`vram // 7` auto-sizer). The actual RunPod image auto-sizer in
`infra/runpod-marker/run.sh` divides by 8 GiB (`vram_mb / 8192`) and then caps
at `min(cpu_count, 32)` — that is why the B200 pod sized to 22 workers
(~179 GiB / 8), not 25 (÷7). Any worker count above 32 requires overriding
`MARKER_UVICORN_WORKERS`, since the auto-sizer can never produce it.

If one usable Marker worker needs about 7 GB VRAM including model copy and
runtime headroom:

| GPU | Theoretical workers | Safer starting cap | Notes |
| --- | ---: | ---: | --- |
| H100 80 GB | 11 | 8-10 | Good benchmark baseline; enough VRAM to validate tuning. |
| B200 180 GB | 25 | 20-24 | Current B200 result used 22 workers (run.sh ÷8 GiB sizer) and still hit CPU/PDF limits on full documents. |
| B300 288 GB | 41 | 32 | VRAM supports more, but run.sh caps at 32 and official RunPod config only lists 32 vCPUs; 32+ would need an explicit env override and is unlikely to help. |

Raw rental break-even versus B300:

| Compare | B300 must be faster by at least |
| --- | ---: |
| H100 PCIe official RunPod price | 2.56x |
| H100 SXM official RunPod price | 2.25x |
| B200 official RunPod price | 1.25x |

Decision rule:

- H100 is the next sane baseline if a cheaper pod is available and the demo
  target is page-limited preview OCR.
- B200 is useful for proving whether high VRAM plus more workers helps after
  thread/batch tuning.
- B300 is only worth testing if B200/H100 results show worker count or VRAM is
  the limiter.
- A cheaper 4090/L40S-class pod may be worth testing for preview mode if the
  product can tolerate lower concurrency.
- Do not judge by VRAM alone. Judge by effective pages/sec, docs/min, p95
  latency, failure rate, output quality, and cost per successful imported
  course.

### Known Marker Gotchas

- Do not restart uvicorn inside the normal custom image unless you intend to
  restart the pod. The benchmark helper defaults `MANAGE_MARKER=0`.
- RunPod SSH may require `ssh -tt`.
- The helper script can look noisy because base64 payloads and heredocs echo in
  the RunPod PTY.
- `RUNPOD_START_SERVICES=1` can trigger RunPod startup service logs such as apt
  update output; this is normal for the observed base image path.
- Tokens are generated or supplied per pod. Do not bake a real token into the
  image.
- Request concurrency is not the same as uvicorn worker count.
- Full-document imports are long-tail heavy. One nasty PDF can dominate a batch.

- The best concurrency is selected from measured throughput and p95, not
  assumed from worker count.
- Oghma env is pointed at the pod with `MARKER_PAGE_RANGE=0-2` only for a

## Workstream 2: Chat Stream Lifecycle

Status update 2026-07-03: this workstream is already committed as
`28b67dd fix: harden chat streaming lifecycle` (2026-06-25) on `dev`. The
description below is kept for context; the open decisions about `done`
strictness and logging verbosity still stand.

Files:

```text
src/app/api/chat/route.ts
src/lib/chat/hooks/use-chat-stream.ts
src/lib/chat/hooks/use-chat-persistence.ts
src/lib/chat/parse-sse-frame.ts
src/__tests__/lib/chat/parse-sse-frame.test.ts
```

Intent:

- Make streaming chat failures and browser disconnects observable.
- Distinguish a clean stream finish from a truncated stream.
- Preserve partial assistant draft content when the user unloads or hides the
  tab mid-stream.

Server-side changes:

- Creates a per-stream `chatStreamId`.
- Tracks elapsed time, bytes sent, last SSE event, and client disconnect state.
- Wraps the `ReadableStream` controller in a safer writer that ignores writes
  after disconnect/close and logs failed enqueues.
- Logs stream start, successful completion paths, fallback completion,
  tool-call-limit completion, empty-response fallback, interruption, and
  failures.
- Implements `cancel(reason)` to record client disconnects.
- Sends explicit `done` SSE events as stream completion markers.

Client-side changes:

- `parseSseFrame` now returns `{ type: "done" }` for `event: done`.
- `use-chat-stream` tracks whether `done` was seen.
- If the response body ends without a `done` event, the hook logs a warning and
  throws `Response stream ended before completion`.
- Adds client console logging for request start, completion, aborts, stream
  failures, metadata receipt, and done receipt.
- `use-chat-persistence` adds debug logging around draft-save decisions and
  saves partial assistant drafts on `beforeunload` / hidden visibility while
  streaming.

Tests:

- `src/__tests__/lib/chat/parse-sse-frame.test.ts` adds coverage for parsing
  `done` lifecycle events.

Risks/checks:

- Requiring `done` is intentionally stricter. If any proxy, runtime, or
  fallback path closes without `done`, users will now see a stream error instead
  of silent partial completion.
- Console logging is useful for diagnosis but may be noisy in production if
  left at current verbosity.
- Recommended verification: run chat streaming manually, interrupt/refresh
  mid-stream, and run the chat parser tests.
