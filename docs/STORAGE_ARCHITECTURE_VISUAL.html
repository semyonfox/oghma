🏗️ OghmaNotes Storage Architecture

Consolidated PostgreSQL-Primary Storage with BullMQ OCR/RAG Pipeline

The Current Problem

❌ Before: Dual Storage

  • Notes in PostgreSQL AND S3
  • Tree structure in PostgreSQL AND S3
  • Settings only in S3
  • One-way sync (S3→PG) incomplete
  • No clear source of truth
  • Data inconsistency risks
  • Complex debugging

✅ After: Single Source of Truth

  • PostgreSQL = master
  • S3 = binaries only
  • No sync complexity
  • Single query answers questions
  • Consistent state always
  • Audit trail via timestamps
  • Clear responsibilities

Storage Distribution

Data Type Primary Backend Why Here? Example
Note Metadata PostgreSQL Indexed queries, consistency title, timestamps, flags
Note Content PostgreSQL Edited in Lexical, full-text search HTML/Markdown
Tree Structure PostgreSQL Hierarchical queries, single source folder hierarchy, parent_id
Extracted Text PostgreSQL Full-text search, processed OCR output (cleaned)
Embeddings PostgreSQL (pgvector) Semantic search, RAG retrieval 1536-dim vectors
Binary Attachments S3 Large files, CDN, cost-effective PDFs, images, videos
Raw OCR Output S3 Audit trail, large text Full pre-processed OCR

Data Flow: Upload → Searchable

Core Components

PostgreSQL (Source of Truth)

  • All note metadata & content
  • Tree structure & hierarchy
  • Extracted text (processed)
  • pgvector embeddings
  • Audit timestamps
  • Processing status

S3 Storage (Assets)

  • Binary files (PDFs, images)
  • Raw OCR outputs
  • Signed URLs (on-demand)
  • Archive & versioning
  • Cost-effective blob storage
  • Disaster recovery

BullMQ + Redis

  • Async job queue
  • OCR worker pool
  • Embedding generation
  • Retry logic & backoff
  • Job monitoring
  • Dead letter handling

Search & RAG

  • Full-text search (tsvector)
  • Semantic search (vectors)
  • Similarity ranking
  • RAG context retrieval
  • Progress tracking
  • Error recovery

API Endpoints (Updated)

Endpoint Method Auth Purpose
/api/upload POST ✓ Required Upload file → S3 + enqueue OCR
/api/notes/{id} GET ✓ Required Fetch note (from PostgreSQL)
/api/notes/{id} PUT ✓ Required Update note content/metadata
/api/search POST ✓ Required FTS or semantic search
/api/notes/{id}/processing-status GET ✓ Required Get OCR + embedding status
/api/tree GET/POST ✓ Required Tree ops (PostgreSQL only)

Benefits Summary

✅ Single Source of Truth

PostgreSQL is the master. One query answers any question. No more "which one is correct?"

✅ No Sync Complexity

Remove one-way S3→PG sync code. No more stale data or data inconsistency.

✅ Consistent State

Transactions ensure integrity. Audit trail via timestamps.

⚡ Scalable Pipeline

BullMQ worker pool handles OCR asynchronously with retry logic.

🔍 Dual Search

Full-text search (keyword) + semantic search (meaning-based).

🤖 RAG Ready

Embeddings + retrieved text ready for LLM context injection.
