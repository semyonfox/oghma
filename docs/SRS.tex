\section{Introduction}
%==============================================================================

OghmaNotes is a web-based learning platform for students to take markdown notes, upload and extract text from PDFs, search their materials, and ask AI questions grounded in their own course content (Retrieval-Augmented Generation).

\subsection{Purpose}

This document defines the scope, architecture, and features for OghmaNotes. It reflects the codebase as deployed in April 2026.

\subsection{Audience}

Development team, supervisors, and module assessors.

\section{Scope}
%==============================================================================

\subsection{In Scope}

\subsubsection{Core Learning System}

\begin{itemize}
    \item \textbf{Note Taking:} Markdown editor with offline support (PWA), auto-save, conflict resolution
    \item \textbf{Note Organization:} Hierarchical folder structure with drag-and-drop support
    \item \textbf{File Storage:} AWS S3 for notes and files
    \item \textbf{PDF Upload:} Store PDFs up to 100MB per file
    \item \textbf{PDF Annotations:} Page-level annotation storage linked to attachments
\end{itemize}

\subsubsection{Search System}

\begin{itemize}
    \item \textbf{Fuzzy Keyword Search:} Full-text search across notes and PDFs
    \item \textbf{Semantic Search:} Vector-based similarity search using embeddings
    \item \textbf{Search UI:} Command palette (Cmd+K) with filters and snippets
    \item \textbf{Note Extraction:} Text extraction from PDFs with metadata preservation
\end{itemize}

\subsubsection{AI-Powered Features}

\begin{itemize}
    \item \textbf{RAG Chat:} Users can ask questions about their notes and PDFs and receive AI answers with cited source material
    \item \textbf{PDF Chunking:} Automatic text segmentation (approximately 500 characters per chunk, sentence-aligned with no overlap)
    \item \textbf{Vector Embeddings:} 1024-dimensional embeddings via an external embedding API
    \item \textbf{Reranking:} External reranking API for relevance precision
    \item \textbf{Streaming Responses:} Token-by-token response generation
    \item \textbf{Source Citations:} Clickable links to source PDF locations and notes
    \item \textbf{Chat Sessions:} Persistent chat history with per-message source attribution and rating
\end{itemize}

\subsubsection{Learning Tools}

\begin{itemize}
    \item \textbf{Quiz Generation:} Auto-generate quizzes from notes and PDFs
    \item \textbf{Quiz Types:} Multiple choice, short answer, true/false
    \item \textbf{Bloom Levels:} Questions tagged with Bloom's taxonomy level
    \item \textbf{Flashcard System:} Spaced repetition using the FSRS (Free Spaced Repetition Scheduler) algorithm via \texttt{ts-fsrs}
    \item \textbf{Review History:} Per-review tracking including response time and AI-generated feedback
    \item \textbf{Study Streaks:} Daily review streak tracking with milestone records
\end{itemize}

\subsubsection{Canvas LMS Integration}

\begin{itemize}
    \item \textbf{API Token Connection:} Secure encrypted token storage for Canvas REST API access
    \item \textbf{Assignment Sync:} Auto-import assignments and due dates
    \item \textbf{Course Settings:} Per-course activation, archival, and display preferences
    \item \textbf{File Import:} Async Canvas file import with per-file status tracking
    \item \textbf{Daily Sync:} Automatic sync of new assignments with a manual refresh option
\end{itemize}

\subsubsection{Calendar, Planning \& Assignment Tracking}

\begin{itemize}
    \item \textbf{Assignment Calendar:} View Canvas assignments and due dates
    \item \textbf{Time Blocks:} User-defined study blocks linked to assignments
    \item \textbf{Pomodoro Sessions:} Pomodoro-style focus sessions linked to time blocks or assignments
    \item \textbf{Calendar Export:} Per-user calendar export token for external calendar subscription
    \item \textbf{Integration with Study Tools:} Link quizzes and flashcards to assignments

\begin{itemize}
    \item AWS S3 for object storage

\begin{itemize}

\begin{itemize}
    \item AWS Amplify Gen 2 for auto-deployment from the main branch
    \item AWS ECS Fargate for the Canvas import worker (Docker containerised)
    \item AWS SQS for the asynchronous job queue
    \item AWS SES for transactional email
    \item AWS ElastiCache (Valkey, latest AWS-managed version) for session caching and rate limiting
    \item AWS EC2 Auto Scaling Group for GPU-backed OCR
    \item AWS RDS PostgreSQL with pgvector
\end{itemize}

\section{Functional Requirements}
%==============================================================================

\subsection{User Authentication}

Users register with email and password, verify their email before first login, and authenticate via either credentials or Google/GitHub OAuth. Passwords are hashed with bcrypt (minimum 10 rounds) and sessions are issued as 24-hour JWTs. Repeated failed logins are rate-limited, and password reset happens via single-use email tokens valid for one hour. Users can delete their account at any time; deletion is soft with a 30-day GDPR-compliant recovery window.

\subsection{Note Taking}

Authenticated users can create, read, update, and delete markdown notes, organised in a drag-and-drop folder hierarchy with pin and soft-delete support. Notes auto-save to IndexedDB every five seconds and sync to S3 on reconnection; concurrent edits resolve last-write-wins. Timestamps (created, updated) and note relationships (parent folder, clone source) are preserved.

\subsection{PDF Upload and Storage}

Users can upload PDF files up to 100MB. Files are validated, stored in S3, and served via short-lived presigned URLs (1-hour expiry). The ingestion pipeline extracts text, captures metadata (filename, page count, MIME type, size), and supports page-level annotations. Upload progress and failure states are surfaced in the UI.

\subsection{Search}

Search combines fuzzy keyword and semantic vector modes over notes and PDFs, surfaced through a Cmd+K command palette. Results include relevance scores, text snippets, and filters for document type and date range, with pagination for large result sets.

\subsection{RAG Chat and AI Features}

User questions are answered against their own content. The pipeline chunks source text (approximately 500 characters, sentence-aligned, no overlap), embeds each chunk (1024 dimensions) via an external embedding API, and stores the vectors in pgvector. On query, the system performs HNSW similarity search followed by external reranking, then generates a streamed LLM response with clickable citations back to source material. Chat sessions and messages are persisted, including per-message source attribution and optional user ratings. All provider API keys live on the server, and failures in any external API degrade gracefully rather than crash the chat flow.

\subsection{Quiz Generation and Flashcards}

Quizzes are generated from selected notes or PDF sections as multiple choice, short answer, or true/false questions, each tagged with a Bloom's taxonomy level. Quiz sessions record response time, correctness, and Bloom distribution, and can optionally be timed.

Flashcards are created from quiz questions and scheduled using the FSRS algorithm. Users rate each card Again/Hard/Good/Easy, which updates persisted FSRS state (stability, difficulty, reps, lapses, due date). A daily review queue is derived from due dates, every review is recorded for analytics, and study streaks (current, longest, total days) are tracked.

\subsection{Canvas LMS Integration}

Users connect to Canvas via an encrypted API token, and can disconnect or reconnect at any time. Assignments and due dates sync daily and on demand, with per-course settings controlling which courses are active or archived. File imports run asynchronously through an SQS queue and an ECS Fargate worker: per-file status is tracked, imported PDFs flow through Marker OCR and the embedding pipeline, and sync history is preserved across jobs. API failures are surfaced without aborting the rest of the sync.

\subsection{Assignment Calendar and Planning}

Canvas assignments appear in a calendar with week and month views and due-date warnings (for example, \emph{Due in 3 days}). Users can schedule study time blocks against assignments and run Pomodoro sessions linked to either. A per-user calendar export token allows subscribing to the calendar from external apps.

\section{Database Design}
%==============================================================================

Data is stored in PostgreSQL (AWS RDS) with the pgvector extension enabled for semantic search. All primary keys use UUID v7. Individual table columns are expected to evolve during iteration and are not specified in advance; this section outlines the core entities and the relationships between them.

\subsection{Core Entities}

\begin{itemize}
    \item \textbf{User} (\texttt{login}) --- email, hashed password, verification state, soft-delete flag. Extended over time with profile fields, Canvas connection data, and reset/verification tokens.
    \item \textbf{OAuth account} --- provider records linked to a user for Google and GitHub sign-in.
    \item \textbf{Note} --- title, markdown content, S3 key for the canonical JSON, folder flag, pin state, soft-delete. Owned by a user.
    \item \textbf{Tree item} --- positions notes and folders inside the user's hierarchy; self-referential parent link.
    \item \textbf{Attachment} --- files uploaded against a note (primarily PDFs): filename, S3 key, MIME type, size.
    \item \textbf{Chunk} --- a passage of text extracted from a note or attachment for retrieval.
    \item \textbf{Embedding} --- a 1024-dimensional vector associated one-to-one with a chunk. Kept in a separate table so embeddings can be regenerated without rewriting chunk rows.
    \item \textbf{Chat session and message} --- persistent RAG chat history with per-message source attribution.
    \item \textbf{Assignment} --- Canvas-synced assignment with title, course, and due date.
    \item \textbf{Canvas import job} --- top-level job record and per-file import records, populated by the async worker.
    \item \textbf{Quiz question} --- generated question (multiple choice, short, true/false) linked to a source note or chunk.
    \item \textbf{Quiz card} --- FSRS state per question per user (stability, difficulty, due date, reps, lapses).
    \item \textbf{Time block and Pomodoro session} --- planning primitives linked to assignments.
\end{itemize}

Supporting tables for ingestion jobs, review history, rate limiting, streaks, and per-course settings are expected to emerge during iteration as the corresponding features are built.

\subsection{Key Relationships}

\begin{itemize}
    \item Every user-scoped row carries a \texttt{user\_id} foreign key to \texttt{login}, so all queries can be filtered at the row level for isolation
    \item A note has many attachments, chunks, and (through chunks) embeddings
    \item Tree items form a per-user forest via a self-referential \texttt{parent\_id}
    \item A chat session belongs to a user and optionally a note; messages belong to a session
    \item Canvas import jobs have many per-file import records; each import can link to the note it produced
    \item Quiz cards sit between a question and the FSRS review history, so the same question can be reviewed repeatedly over time
\end{itemize}

\subsection{Indexing Strategy}

\begin{itemize}
    \item HNSW index on the embedding vector column for approximate nearest-neighbour search
    \item GIN full-text index on note content and extracted text for fuzzy keyword search
    \item Compound \texttt{(user\_id, created\_at)} and \texttt{(user\_id, updated\_at)} indexes across user-scoped tables
    \item Targeted indexes on review due dates and assignment due dates for the review queue and calendar views
\end{itemize}

\section{Design Decisions}
%==============================================================================

\subsection{Server-Managed LLM Integration}

\textbf{Decision:} LLM, embedding, and reranking API keys are managed server-side. Users do not need to provide their own keys.

\textbf{Rationale:}
\begin{itemize}
    \item No API key setup required for users
    \item Consistent model selection across all users
    \item Centralised cost monitoring and usage tracking
    \item Providers can be swapped without any user-facing change (and were, multiple times, during iteration)
\end{itemize}

\textbf{Implementation:}
\begin{itemize}
    \item All inference requests go through backend API routes
    \item Provider abstraction allows embedding and reranking vendors to be swapped via configuration
    \item BYOK (Bring Your Own Key) is planned as a future optional feature
\end{itemize}

\subsection{Note Storage Architecture}

\textbf{Decision:} Notes stored in S3 as JSON files, not in PostgreSQL BLOB columns.

\textbf{Rationale:}
\begin{itemize}
    \item S3 scales independently from the database
    \item Simpler backup and disaster recovery
    \item Better versioning and audit trail support
    \item Works well with PWA offline storage
\end{itemize}

\subsection{UUID v7 for All IDs}

\textbf{Decision:} All primary keys use UUID v7 (not auto-increment integers).

\textbf{Rationale:}
\begin{itemize}
    \item Cryptographically secure (no ID enumeration attacks)
    \item Sortable by timestamp (useful for time-series queries)
    \item No central coordination needed for distributed systems
    \item Zero collision risk
\end{itemize}

\subsection{Semantic Search First}

\textbf{Decision:} Semantic vector search is the primary search mode; fuzzy search is a fallback.

\textbf{Rationale:}
\begin{itemize}
    \item Better user experience (understands meaning, not just keywords)
    \item Integrates naturally with RAG chat
    \item Single-mode search UI (less cognitive load)
\end{itemize}

\subsection{No Collaborative Editing}

\textbf{Decision:} Notes are single-user only (no real-time collaboration).

\textbf{Rationale:}
\begin{itemize}
    \item Simplifies architecture (no WebSockets, CRDT, conflict resolution)
    \item Reduces scope and development time
    \item Sufficient for individual student use case
    \item Can be added later if needed
\end{itemize}

\subsection{Hosted in AWS Ireland}

\textbf{Decision:} All infrastructure is deployed to the \texttt{eu-west-1} (Ireland) region.

\textbf{Rationale:}
\begin{itemize}
    \item Low latency for the primary user base (Irish university students)
    \item EU data residency for GDPR alignment
    \item Single region simplifies networking and observability
\end{itemize}

\section{Glossary}
%==============================================================================

\begin{description}
    \item[RAG] Retrieval-Augmented Generation. Technique for answering questions by retrieving relevant documents and using an LLM to synthesize answers.
    \item[UUID v7] Universally Unique Identifier version 7. Cryptographically secure, sortable by timestamp.
    \item[pgvector] PostgreSQL extension for storing and querying vector embeddings.
    \item[Semantic Search] Search based on meaning using vector embeddings, not keyword matching.
    \item[FSRS] Free Spaced Repetition Scheduler. Modern spaced repetition algorithm with improved accuracy over SM-2, implemented via the \texttt{ts-fsrs} library.
    \item[HNSW] Hierarchical Navigable Small World. Graph-based index structure for approximate nearest-neighbour search in high-dimensional vector spaces.
    \item[SQS] AWS Simple Queue Service. Managed message queue for decoupling application components.
    \item[ECS Fargate] AWS Elastic Container Service with Fargate launch type. Serverless container runtime.
    \item[PWA] Progressive Web App. Web application that works offline and can be installed.
    \item[JWT] JSON Web Token. Stateless authentication mechanism.
    \item[S3] AWS Simple Storage Service for object storage.
    \item[Valkey] Open-source fork of Redis; used here via AWS ElastiCache for session caching and rate limiting.
    \item[Bloom Level] Level in Bloom's taxonomy of cognitive objectives, used to classify quiz questions by difficulty type.
\end{description}
