\section{Project Overview}

\subsection{Problem Statement}

University students' course content is scattered across LMS platforms, local folders, and personal notes. Existing tools like Notion and Obsidian offer strong editors but treat uploaded documents as opaque files --- users cannot query their content semantically. Dedicated AI tools can answer questions but lack persistent knowledge bases scoped to a student's own materials. No existing tool bridges the gap between a personal note-taking workspace and AI-powered retrieval over lecture content, integrated with the institution's LMS.

\subsection{Our Solution}

OghmaNotes is an AI-powered note-taking platform purpose-built for students. It provides a unified workspace where users can create and organise notes using a VSCode-inspired editor, import course materials directly from Canvas LMS, and query their entire knowledge base through a Retrieval-Augmented Generation (RAG) chat interface.

The platform is built with Next.js~16.2 on the frontend, a PostgreSQL database with the \tech{pgvector} extension for vector similarity search, and a suite of AWS services (RDS, S3, SQS, ECS Fargate, SES, Amplify) for production-grade infrastructure. It is live at \url{https://oghmanotes.ie}.

\subsection{Key Differentiators}

Key differentiators include: (1)~RAG-powered search over notes and uploaded PDFs using semantic retrieval rather than keyword matching; (2)~direct Canvas LMS integration with one-click course import processed by a self-scaling worker; and (3)~a full AWS production stack demonstrating real-world cloud engineering. The application supports 12 languages through a custom internationalisation system.

\subsection{Authentication}

OghmaNotes supports email/password registration (with SES-based email verification and password reset) and OAuth~2.0 via Google and GitHub using Auth.js. Accounts are deduplicated by email, and rate limiting is applied to all auth endpoints. The full authentication flow is detailed in Section~6.1.

\subsection{Notes Editor}

The editor builds on the Notea foundation (S3-backed markdown storage), modernised and extended as described in Pivot~1. The interface follows a VSCode-inspired three-panel layout: a file tree sidebar (Obsidian-style with indent guides), a main editor pane (Lexical rich-text with CodeMirror for markdown source), and an optional split-pane PDF viewer for side-by-side study. File management includes create, rename, delete, duplicate, drag-and-drop upload, breadcrumb navigation, and a keyboard-accessible command palette.

\subsection{Canvas LMS Integration}

Users provide their Canvas API token and institution URL in settings. The token is encrypted at rest and decrypted at runtime for Canvas REST API calls. Users then browse courses and select which to import; the process runs asynchronously via SQS and an ECS Fargate worker (see Section~6.2).

\screenshotplaceholder{Canvas Import --- Settings page with API token configuration}

\screenshotplaceholder{Canvas Import --- Course selection and import progress}

\subsection{Search}

Notes and imported documents are searchable via two complementary modes. Keyword search performs fast pattern matching against note titles and content. Semantic search embeds the query using \tech{mxbai-embed-large} and retrieves the most relevant notes by cosine similarity via pgvector. Both modes support filtering by Canvas course, allowing students to scope searches to a specific module. The two modes run as separate requests and results are deduplicated client-side.

\subsection{AI Chat (RAG-Powered)}

Users ask natural-language questions about their notes and imported documents. The system embeds the query, retrieves relevant chunks via pgvector, reranks them, and streams a cited answer from the LLM. Users can scope search to specific files or folders, chat sessions persist across visits, and responses can be rated with a thumbs up or down. The full technical pipeline is detailed in Section~6.3.

\subsection{Settings}

The settings page provides profile management, dark mode, language selection (13 locales), Canvas API token configuration, OAuth account linking, and full vault export. The vault export streams all notes as Markdown files in a ZIP archive, preserving the folder hierarchy, with no proprietary format lock-in.

\subsection{Quiz System (Spaced Repetition)}

OghmaNotes includes a flashcard-based quiz system that uses the FSRS (Free Spaced Repetition Scheduler) algorithm via the \tech{ts-fsrs} library. Users can create flashcards from their notes and study them with scientifically-backed scheduling that optimises long-term retention. The system tracks quiz sessions, individual card reviews, and maintains streak data to encourage consistent study habits.

\section{REST API}

The backend exposes 16 REST API route groups under \tech{/api/} via Next.js API Routes, covering authentication, notes CRUD, file operations, Canvas import management, AI chat, search, quiz/flashcard management, productivity features, and data portability.

\subsection{Authentication Middleware}

All protected routes use a custom JWT middleware that extracts the token from HTTP-only cookies, verifies the signature, and attaches the decoded user object to the request context. Unauthenticated requests receive a 401 response. Rate limiting is implemented using a sliding-window counter pattern, with stricter limits on authentication endpoints (login, register, password reset) to prevent brute-force attacks.

\subsection{RAG Query Flow}

The query is embedded (self-hosted \tech{mxbai-embed-large}, 1024d), compared against stored chunks via pgvector cosine search (top-20 within distance 0.75, optionally scoped to selected files), reranked by a self-hosted cross-attention model (retaining top-5 above a 0.15 relevance floor), injected into a prompt, and streamed to the user from Kimi K2.5 via SSE. Messages and responses are persisted for session continuity. Thresholds were tuned empirically against university lecture queries.

\section{CI/CD \& Deployment}

\subsection{AWS Amplify Pipeline}

The project uses AWS Amplify Gen~2 as the primary CI/CD platform. The build and deployment pipeline is triggered automatically on each push to the \tech{main} branch. The pipeline executes the following stages:

\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Install:} Runs \tech{npm ci} to install dependencies from the lockfile.
    \item \textbf{Build:} Executes \tech{next build}, producing optimised static assets and serverless functions.
    \item \textbf{Deploy:} Distributes the build output to Amplify's CDN and deploys SSR routes as Lambda@Edge functions.
\end{enumerate}

Environment variables (database credentials, API keys, AWS service endpoints) are managed through Amplify's environment configuration, with separate variable sets for the \tech{dev} and \tech{main} branches.

\subsection{Branch Strategy}

The repository follows a two-branch model. The \tech{dev} branch serves as the integration branch where features are developed and tested. When stable, changes are merged to \tech{main} via pull request, triggering the production deployment. Both branches have corresponding Amplify environments with independent configurations.

\subsection{Containerised Infrastructure}

The repository contains three Dockerfiles, each targeting a distinct runtime: the Next.js production application (three-stage Alpine build with standalone output), the Canvas import worker (\tech{Dockerfile.worker}, Node 22 with \tech{tsx}), and the Marker OCR service (\tech{Dockerfile.marker}, Python 3.11 with GPU support). Worker and Marker images are pushed to Amazon ECR; ECS Fargate pulls the worker image on demand, scaling from zero when SQS messages arrive.

A Husky~v9 pre-commit hook enforces quality gates on every commit: full ESLint linting and the complete Vitest test suite (blocking), \tech{Dockerfile.marker} syntax validation via \tech{docker build --dry-run}, and \tech{shellcheck} on all infrastructure shell scripts. For the Marker OCR instance, a custom AMI baking script (\tech{bake-ami.sh}) pre-installs all Python dependencies and the CUDA runtime into a machine image, reducing cold-start time from 5--7~minutes to approximately 90~seconds. The baked AMI is deployed via a Launch Template within an Auto Scaling Group behind an Application Load Balancer, with IAM roles granting ECR pull access.

\section{Design Decisions \& Architectural Pivots}
\label{sec:pivots}

The team made several significant architectural pivots during development, each driven by constraints encountered at runtime.

\subsection{Pivot 1: Notea Fork to Modern Next.js Application}

The project started from Notea, an MIT-licensed, unmaintained open-source note-taking application that stored markdown notes on S3 --- a solid backbone to scaffold from. However, Notea's codebase was built on outdated, unmaintained dependencies across the stack. Modernising it required upgrading through three major React versions (16~$\to$~19), where significant APIs were deprecated or removed, replacing the legacy editor with Lexical and CodeMirror, and swapping unmaintained packages for actively supported alternatives throughout. This upgrade work was substantial but gave the team a production-ready S3-backed markdown foundation to extend with RAG, Canvas integration, and the full AWS infrastructure.

\textbf{Lesson:} Forking saves initial scaffolding time, but inheriting an unmaintained dependency tree creates upgrade debt that compounds with each major framework version.

\subsection{Pivot 2: Monorepo to Single Application}

The project started as a monorepo with separate packages under an \tech{apps/web/} directory. This introduced complexity in the Amplify build configuration that proved disproportionate to the benefits for a three-person team. The team collapsed the structure into a single Next.js application, simplifying both development and deployment.

\subsection{Pivot 3: BullMQ (Redis) to AWS SQS}

The Canvas import job queue was initially implemented using BullMQ backed by ElastiCache (Valkey/Redis). This worked in local development but failed in production: Amplify's Lambda functions could not reach ElastiCache without VPC configuration, and the Amplify Lambda runtime had a 29-second timeout that was incompatible with long-running import jobs. The team migrated to AWS SQS, which uses IAM authentication (no VPC required) and decouples the API from the worker entirely.

\textbf{Lesson:} Cloud networking constraints drive architectural decisions more than theoretical best practices. SQS was the pragmatic choice given the deployment environment.

\subsection{Pivot 4: Local Ollama to Cohere Embeddings}

\begin{sloppypar}
The initial embedding pipeline used a self-hosted Ollama instance running \tech{qwen3-embedding:8b}, served over a Cloudflare Tunnel. While the model itself performed well, the Cloudflare Tunnel introduced network-level instability --- embedding requests during Canvas imports would intermittently time out, causing ingestion jobs to fail mid-batch. Rather than debug the tunnel reliability, the team migrated to Cohere's \tech{embed-multilingual-v3.0} API, which provided a stable external endpoint with consistent latency. As a secondary benefit, Cohere's 1024-dimensional output was also compatible with pgvector's HNSW index limit.
\end{sloppypar}

\textbf{Lesson:} Infrastructure reliability is as important as model quality. A locally-hosted model routed through a tunnelled connection introduces failure modes that a direct API call does not.

\subsection{Pivot 5: OpenWebUI as Secure Model Gateway}

The homelab hosts multiple AI models (embedding, reranking, LLM) that must be accessible to the production application but not to the public internet. OpenWebUI serves as the secure gateway for these models, exposed via a Cloudflare Tunnel with strong authentication and sign-ups locked out, preventing any unauthorised use of the homelab's compute resources. The production application authenticates through OpenWebUI's API to access Ollama-hosted models, while the Kimi K2.5 LLM is accessed directly via its OpenAI-compatible API. This dual-provider pattern allows seamless switching between self-hosted and cloud models via environment variables.

\subsection{Pivot 6: pnpm to npm}

A mid-project package manager switch from pnpm to npm was necessitated by compatibility issues with the Amplify build environment. While pnpm offers faster installs and stricter dependency resolution, its workspace features conflicted with Amplify's build process. The switch to npm resolved build failures with minimal disruption.

\subsection{Pivot 7: Cohere Embeddings Back to Self-Hosted Homelab}

After migrating to Cohere (Pivot~4), the team encountered aggressive rate limiting on Cohere's free-tier embedding API during Canvas imports, where hundreds of chunks required embedding in rapid succession. Rather than upgrade to a paid tier, the team resolved the underlying Cloudflare Tunnel reliability issues and migrated back to the self-hosted \tech{qwen3-embedding:8b} model on the project lead's homelab. This eliminated rate limits and external API costs entirely. With the tunnel now stable, the team also took the opportunity to restore the original 4096-dimensional vectors for higher semantic fidelity, accepting the trade-off of sequential pgvector scanning at current data scale. The reranker was similarly moved to self-hosted infrastructure.

\textbf{Lesson:} External API dependencies introduce constraints beyond cost --- rate limits can silently degrade user experience. Self-hosting critical pipeline components provides control over throughput at the cost of operational responsibility.

\subsection{Pivot 8: qwen3-embedding:8b to mxbai-embed-large (Right-Sizing for Hardware)}

After restoring self-hosted embeddings (Pivot~7), the team discovered a performance bottleneck caused by hardware constraints on the homelab server. The \tech{qwen3-embedding:8b} model required 6.1\,GB of memory, but the server's GPU --- a GTX~1050~Ti Max-Q --- had only 4\,GB of VRAM. Ollama automatically split the model: 59\% ran on the GPU while 41\% fell back to CPU inference on the i7-8750H. This CPU portion became the dominant bottleneck, causing erratic embedding latency that varied with available CPU headroom.

Running the model entirely on CPU would have been worse, losing the GPU acceleration entirely. Instead, the team migrated to \tech{mxbai-embed-large}, a 335M-parameter embedding model that fits entirely within the 4\,GB VRAM budget. The migration required a database schema change from \tech{vector(4096)} to \tech{vector(1024)}, clearing all existing embeddings, and re-embedding all stored content. As a secondary benefit, 1024~dimensions falls within pgvector's HNSW index limit of 2000, enabling approximate nearest-neighbour indexing --- a capability that was previously unavailable with 4096-dimensional vectors.

\textbf{Lesson:} Model selection must account for the full inference environment, not just benchmark quality. A smaller model running entirely on GPU will consistently outperform a larger model partially offloaded to CPU. Right-sizing the model to the available hardware eliminated the erratic latency and improved embedding throughput by approximately 10$\times$.

\subsection{Budget Constraints}

The project operated within an AWS education credits budget. This constraint directly influenced infrastructure decisions: the self-scaling ECS Fargate pattern (scaling to zero when idle) was chosen specifically to minimise compute costs. Instance sizes were kept minimal (\tech{db.t3.micro} for RDS, \tech{cache.t3.micro} for ElastiCache), and services were deployed in a single region (\tech{eu-north-1}) to avoid cross-region transfer costs.

\section{Challenges \& Reflections}

\subsection{Semyon Fox}

As technical lead, my biggest challenge was scope management. The combination of AI-assisted development and genuine ambition created a feedback loop: every feature felt achievable, so we kept adding them. The project grew from a simple notes app to a full-stack platform with RAG, Canvas integration, spaced repetition, and a GPU-backed OCR pipeline --- all within a university project timeline.

The most technically demanding work was the AWS infrastructure. Debugging why Amplify's Lambda functions couldn't reach ElastiCache (VPC networking), why SQS messages weren't being consumed (IAM permissions), and why Marker's GPU instance took 15 minutes to cold-start (pip installing marker-pdf on every boot, solved by pre-baking a Docker image) each consumed entire days. These problems don't appear in tutorials or documentation --- they emerge from the specific intersection of services in your architecture.

My role was more glue than solo builder. Major architectural decisions were discussed as a group, and AI-assisted development meant features that would otherwise have taken weeks could be scoped and prototyped quickly. That speed was a double-edged sword --- it made the project's ambition feel achievable but compressed the time available for the kind of documentation and interface contracts that would have made integration smoother for everyone.

The key lesson from this project is that architecture decisions made under time pressure have long-lasting consequences. The BullMQ-to-SQS migration, the embedding pipeline iterations (Ollama to Cohere to self-hosted), and the monorepo collapse each cost days of rework. Building a proof-of-concept for each integration point before committing to it would have saved significant time.

\subsection{Samuel Regan}

For me, the most glaring challenge wasn't writing code. It was the learning curve and the invisible layer between the code and the services.

In a project of this magnitude, there were so many components from the AWS services such as instances and environment variables that when something failed even getting the logs for the deployed project was a huge endeavour. Most of the time the code would fail silently and it was almost always not an internal server error, but a configuration issue. For example, I spent an entire day debugging why the rag pipeline worker queue was not dispatching ingestion requests, and when it did it couldn't find the database or persist any data. The root cause was a second database instance running in parallel. My IDE was pointed to a different, older one, so the SELECT checks always returned empty chunk columns.

The ambition for this project between myself and my teammates compounded this. This was our first chance to develop a real full stack app alongside our course, and thus we had chosen the technologies and methodologies we believed would be most suitable for the project and consequently, I was overwhelmed with a long list of technologies, such as TypeScript, Next.js, and a full array of AWS services, as well as learning team Git workflows for the first time which all formed into a very harsh learning curve.

Upon reflection, I have learnt that on a production level, programming skill and familiarities with the technologies are equally important. For the first half of this project I neglected the services making the production possible and focused on the backend code, and that tunnel vision backfired greatly when I tried to integrate my features into the rest of the system.
That said, thanks to this ambitious project I have become far more familiar with cybersecurity, the connection between frontend and backend functions/services such as features being too slow for a UI facing application,integrating LLM models and manipulating them for niche purposes, the services required to host an application on AWS, and the critical thinking required to invent my own solutions to problems, such as our RAG pipeline ingesting too slowly.
The very worker queue I described debugging earlier was my solution to that problem, processing documents in an ordered queue with workers rather than piling on large datasets to be ingested at once.


\subsection{Shreyansh Singh}

Working on OghmaNotes was my first experience with production cloud infrastructure. Setting up AWS RDS with SSL certificates, configuring S3 bucket policies, and getting SES verified for transactional email each presented their own learning curves. The most challenging aspect was understanding how the various AWS services connect --- security groups, IAM roles, and VPC configurations are not intuitive when coming from a local development background.

The OAuth implementation with Auth.js required careful attention to the callback flow and account linking logic, particularly handling edge cases where a user might register with email and later try to sign in with Google using the same email address. The Canvas API integration involved navigating a large REST API surface and understanding the Canvas data model (courses, modules, items, files), which required significant research and experimentation.

The key takeaway from this project was that real-world software engineering is as much about infrastructure and integration as it is about writing application code. The skills I developed in AWS, database design, and API integration are directly applicable to professional work.

\section{Future Work}

\textbf{Hybrid Search:} Combining vector similarity with BM25 keyword search would improve retrieval for queries containing technical terms or proper nouns that benefit from exact matching alongside semantic understanding.

\textbf{Note Tagging and Bidirectional Linking:} Adding arbitrary tags to notes and supporting wiki-style \texttt{[[Note Title]]} links with backlink display and a knowledge graph visualisation would improve note discoverability and reflect the original SRS scope.

\textbf{Analytics Dashboard:} A dedicated dashboard showing study time trends, quiz score history, and weak topic identification would give students deeper insight into their learning patterns. The current quiz dashboard exposes a summary widget (streak, mastery percentage, cards due today); full time-series analytics remain a planned enhancement.

\textbf{Auto-Scheduling Study Sessions:} Automatically generating time blocks in the calendar based on Canvas assignment due dates and FSRS review schedules would close the loop between the LMS integration and the productivity tools.

\textbf{Collaborative Editing:} Real-time collaborative editing would allow study groups to work on shared notes simultaneously, integrated with the RAG system.

\textbf{Expanded LMS Support:} Adding integrations for Moodle, Blackboard, and other learning management systems would broaden the platform's applicability beyond Canvas.

\textbf{BYOK (Bring Your Own Key):} Allowing students to connect their own LLM API keys would provide model choice and reduce platform costs. The settings UI stub already exists.

\textbf{Offline Support (PWA):} Implementing a Progressive Web App with IndexedDB caching would allow students to review notes and flashcards without an internet connection, syncing changes when reconnected.

\section{AI-Assisted Development Disclosure}

This project used AI coding assistants --- primarily Claude Code (Anthropic), with supplementary use of OpenCode and Codex --- throughout development. This section describes the relationship between the developers and these tools.

\subsection{How AI Was Used}

AI served as an implementation accelerator. The developers defined all architectural decisions, feature priorities, and design constraints; AI tools generated code against those specifications, which was then reviewed, tested, and modified before committing. Common use cases included generating boilerplate for repetitive patterns (API routes, database migrations, i18n translation files), debugging complex AWS configuration issues, and UI refinement.

\subsection{What AI Did Not Do}

AI did not make architectural decisions. Every pivot documented in Section~\ref{sec:pivots} --- from VPC networking failures to embedding model right-sizing --- was diagnosed by the developers through production logs, hardware profiling, and AWS documentation. AI tooling generated code; humans debugged it, fixed what broke in production, and made the engineering judgment calls that shaped the system. The process of diagnosing failures AI could not see (cloud networking, GPU memory splits, rate-limit edge cases) proved to be the most valuable learning experience of the project.

\subsection{Reflection}

AI-assisted development shifts the bottleneck from ``can I implement this?'' to ``should I implement this now?'' The ability to ship features quickly made scope management the hardest discipline. The key learning: AI is a force multiplier for execution, but architecture, debugging, and constraint navigation remain fundamentally human engineering skills.
