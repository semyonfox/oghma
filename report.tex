\section{Project Overview}

\subsection{Problem Statement}

University students face a persistent problem with managing lecture materials. Course content is scattered across Learning Management Systems (LMS), local folders, and personal notes. Existing note-taking tools such as Notion or Obsidian offer strong editors, but none combine a notes workspace with AI-powered search over lecture PDFs, nor do they integrate directly with university platforms like Canvas LMS.

Students frequently need to search through dozens of lecture PDFs to find a specific concept, cross-reference notes with slides, or quickly locate an assignment brief buried in a Canvas module. Current workflows involve manually downloading files, switching between applications, and relying on basic keyword search --- which fails when the terminology in notes differs from the source material.

The market for student productivity tools is crowded yet underserved in a critical dimension. Notion, Obsidian, and Google Docs offer strong editing experiences but treat uploaded documents as opaque files --- users cannot query their content semantically. Dedicated AI tools like ChatGPT or Perplexity can answer questions but lack persistent knowledge bases scoped to a student's own course materials. No existing tool bridges the gap between a personal note-taking workspace and AI-powered retrieval over university lecture content, integrated with the institution's own LMS.

\subsection{Our Solution}

OghmaNotes is an AI-powered note-taking platform purpose-built for students. It provides a unified workspace where users can create and organise notes using a VSCode-inspired editor, import course materials directly from Canvas LMS, and query their entire knowledge base through a Retrieval-Augmented Generation (RAG) chat interface.

The platform is built with Next.js~16.2 on the frontend, a PostgreSQL database with the \tech{pgvector} extension for vector similarity search, and a suite of AWS services (RDS, S3, SQS, ECS Fargate, SES, Amplify) for production-grade infrastructure. It is live at \url{https://oghmanotes.ie}.

\subsection{Key Differentiators}

OghmaNotes distinguishes itself from existing tools in several ways. First, RAG-powered search allows users to ask natural-language questions about their notes and uploaded PDFs, with the system retrieving semantically relevant passages rather than relying on exact keyword matches. Second, direct Canvas LMS integration enables one-click import of courses, modules, files, and assignments, processed in the background by a self-scaling worker. Third, the platform runs on a full AWS production stack, demonstrating real-world cloud engineering rather than managed PaaS solutions. The project involved over 840 commits across 69 active development days, with the application supporting 12 languages through a custom internationalisation system.

\subsection{Authentication}

OghmaNotes supports two authentication paths: traditional email/password registration and OAuth via third-party providers.

\subsubsection{Email/Password Authentication}

Users can create an account with an email address and password. The registration flow includes email verification via AWS SES, with a verification link sent to the user's inbox. Login uses JWT-based sessions with secure, HTTP-only cookies. Password reset is available via a token-based email flow, also powered by SES.

\subsubsection{OAuth Authentication}

The application supports OAuth 2.0 login via Google and GitHub, implemented using Auth.js (the successor to NextAuth.js). Users who register with email/password can subsequently link OAuth providers to their account from the settings page. The OAuth implementation handles account deduplication --- if a user signs in with Google using the same email as an existing account, the accounts are linked automatically.

Rate limiting is applied to all authentication endpoints to prevent brute-force attacks.

\subsection{Canvas LMS Integration}

One of OghmaNotes' key differentiators is direct integration with Canvas LMS, the learning management system used by many universities.

\subsubsection{Setup}

Users provide their Canvas API token and institution URL in the settings page. The token is encrypted and stored as the \tech{canvas\_token} column on the \tech{app.login} table, alongside the \tech{canvas\_domain}. The encrypted token is decrypted at runtime to authenticate API requests to the Canvas REST API.

\subsubsection{Import Process}

Once connected, users can browse their Canvas courses and select which ones to import. The import process runs asynchronously:

\begin{enumerate}[leftmargin=1.5em]
    \item The frontend sends an import request to the API, which creates a job record in PostgreSQL and publishes a message to AWS SQS.
    \item An ECS Fargate worker picks up the SQS message and begins processing.
    \item The worker retrieves courses, modules, files, and assignments from the Canvas API.
    \item Downloaded files are uploaded to S3, with metadata stored in PostgreSQL.
    \item PDFs undergo OCR/text extraction (via Marker API with \tech{pdf-parse} fallback), chunking, and Cohere embedding generation.
    \item Embeddings are stored in pgvector for RAG search.
    \item The worker self-scales to zero after an idle period, minimising costs.
\end{enumerate}

Real-time progress is communicated back to the user through status polling and toast notifications.

\subsection{AI Chat (RAG-Powered)}

The AI chat interface allows users to ask natural-language questions about their notes and imported documents.

\subsubsection{How It Works}

When a user sends a message, the system generates an embedding of the query using Cohere's \tech{embed-multilingual-v3.0} model. This embedding is compared against stored document chunks using pgvector's approximate nearest-neighbour search (HNSW index). The top-$k$ most relevant chunks are retrieved and injected into a prompt, which is sent to the LLM (Kimi K2.5 via Moonshot AI). The response is streamed back to the chat interface.

\subsubsection{Context Scoping}

Users can attach specific files or folders to a chat session to scope the RAG search. This ensures that when studying for a particular module, only materials from that module are included in search results --- improving relevance and reducing noise from unrelated content.

Chat sessions persist across visits, allowing users to return to previous conversations.

\subsubsection{Technical Pipeline}

The RAG pipeline combines several components to deliver accurate, source-grounded answers. User queries are embedded with Cohere's \tech{embed-multilingual-v3.0} model (1024 dimensions) using asymmetric input types: \tech{search\_document} for indexing and \tech{search\_query} for queries, which improves retrieval quality by accounting for the stylistic difference between documents and questions.

Before embedding, all markdown syntax is stripped from note content --- headers, emphasis markers, links, code fences, and HTML tags are removed to prevent formatting tokens from diluting the semantic signal in the vector space. Text is then split into sentence-aligned chunks of approximately 500 characters with no overlap, splitting on sentence boundaries (\tech{/(?<=[.?!])$\backslash$s+/}).

At query time, pgvector's HNSW index performs approximate nearest-neighbour search using the \tech{<=>} cosine distance operator, retrieving up to 20 candidates within a maximum distance of 0.75. These candidates are then reranked by Cohere's \tech{rerank-multilingual-v3.0} model, which applies cross-attention scoring to the query--document pairs. Chunks scoring below a 0.15 relevance threshold are discarded, and the top 5 are retained.

The surviving chunks are grouped by source note in the prompt, and the LLM is instructed to cite sources by note title. The production LLM is Kimi K2.5 (Moonshot AI) accessed via an OpenAI-compatible API, configured with a 2048-token generation limit. Responses are streamed to the client via server-sent events with idle timeout reset to maintain connection liveness.

\subsection{AWS Infrastructure}

The production infrastructure runs in the \tech{eu-north-1} (Stockholm) region. The key services and their roles are:

\textbf{AWS Amplify Gen 2} hosts the Next.js application, providing automatic builds from GitHub pushes, CDN distribution, and server-side rendering via Lambda@Edge. The build pipeline runs on each push to the \tech{main} branch, executing \tech{npm ci}, \tech{next build}, and deploying the output.

\textbf{AWS RDS} runs PostgreSQL~17.4 with the \tech{pgvector} extension enabled. The database uses a \tech{db.t3.micro} instance with SSL-encrypted connections. All application tables live under the \tech{app} schema to separate them from system tables.

\textbf{AWS S3} stores all user-uploaded files, notes content, and Canvas-imported documents. Files are organised by user ID and note ID, with presigned URLs used for secure client-side uploads and downloads.

\textbf{AWS SQS} serves as the message queue for Canvas import jobs. When a user initiates an import, the API publishes a message to the SQS queue containing the job ID and user credentials. This decouples the request-handling latency from the potentially long-running import process.

\textbf{AWS ECS Fargate} runs the Canvas import worker as a containerised service. The worker polls SQS for messages, processes imports (downloading files from Canvas, uploading to S3, running OCR, generating embeddings), and scales to zero when idle to minimise costs.

\textbf{AWS SES} handles transactional email for password reset flows and email verification.

\textbf{AWS ElastiCache} (Valkey 8.2) provides in-memory caching for session data.

\textbf{AWS EC2} runs a GPU instance (g4dn.xlarge or g5.xlarge depending on availability) in \tech{eu-west-1} hosting the Marker OCR server for document text extraction. The instance runs within an Auto Scaling Group (ASG) behind an Application Load Balancer. The ASG scales to zero when idle, with a 4--7 minute cold start when documents need processing. The instance supports PDF, DOCX, DOC, PPTX, PPT, PNG, and JPG formats. If the EC2 instance is unavailable, the system falls back to the Datalab Convert API as an external OCR service.

\section{Key System Flows}

This section presents the critical operational flows within OghmaNotes, illustrating how components interact to deliver key features.

\subsection{Authentication Flow}

The authentication system supports two parallel paths that converge on a shared JWT session mechanism.

\textbf{Email/Password Path:} The user submits credentials to \tech{POST /api/auth/register}, which hashes the password with bcrypt, creates a user record in PostgreSQL, and sends a verification email via SES. Upon email verification, the user can log in via \tech{POST /api/auth/login}, which validates credentials, generates a JWT, and sets it as an HTTP-only cookie.

\textbf{OAuth Path:} The user clicks an OAuth provider button (Google and GitHub), which redirects to the provider's consent screen via Auth.js. On callback, Auth.js verifies the token, checks for an existing account with the same email, and either creates a new user or links the OAuth account to an existing one. A JWT is issued identically to the email/password path.

\textbf{Token Refresh:} JWTs have a configurable expiry. The \tech{/api/auth/me} endpoint checks token validity and issues a refreshed token when needed.

\subsection{Canvas Import Pipeline}

The Canvas import pipeline is the most architecturally complex flow in the system, spanning five AWS services.

\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Trigger:} User clicks ``Import'' in the Canvas settings UI. The frontend calls \tech{POST /api/canvas/import} with selected course IDs.
    \item \textbf{Job Creation:} The API creates a \tech{canvas\_import\_jobs} record in PostgreSQL (status: \tech{queued}) and publishes a JSON message to the SQS queue containing the job ID and encrypted Canvas token.
    \item \textbf{Worker Activation:} The ECS Fargate worker, which polls SQS on a loop, receives the message and begins processing.
    \item \textbf{Canvas API Calls:} The worker uses the Canvas REST API to enumerate courses, modules, files, and assignments. Each file is downloaded to a temporary buffer.
    \item \textbf{S3 Upload:} Downloaded files are uploaded to S3 under a user-specific key prefix, with metadata recorded in the \tech{canvas\_imports} table.
    \item \textbf{Text Extraction:} PDFs (and DOCX/PPTX files) are sent to a self-hosted Marker instance running on an EC2 GPU instance within an Auto Scaling Group. If the EC2 instance is unavailable, the system falls back to the Datalab Convert API. Markdown syntax is stripped before embedding to reduce noise in the vector space.
    \item \textbf{Chunking \& Embedding:} Extracted text is split into sentence-aligned, non-overlapping chunks of approximately 500 characters. Each chunk is sent to Cohere's \tech{embed-multilingual-v3.0} API, producing a 1024-dimensional embedding vector.
    \item \textbf{Vector Storage:} Embeddings are stored in the \tech{note\_chunks} table using pgvector, with an HNSW index for efficient approximate nearest-neighbour search.
    \item \textbf{Completion:} The job status is updated to \tech{complete} (or \tech{failed} with error details). The worker returns to polling and scales to zero after an idle timeout.
\end{enumerate}

\subsection{RAG Query Flow}

When a user sends a message in the AI chat interface, the following pipeline executes:

\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Query Embedding:} The user's message is sent to Cohere's embedding API, producing a 1024-dimensional query vector.
    \item \textbf{Vector Search:} The query vector is compared against stored chunk embeddings using pgvector's \tech{<=>} (cosine distance) operator with an HNSW index. The top-$k$ most similar chunks are retrieved (filtered to a maximum cosine distance of 0.75), optionally scoped to the user's selected context (specific files or folders).
    \item \textbf{Reranking:} Retrieved chunks are passed through Cohere's rerank-multilingual-v3.0 model, which rescores them by relevance to the original query. Chunks below a minimum relevance threshold~(0.15) are discarded. This significantly improves precision over raw vector similarity alone.
    \item \textbf{Prompt Construction:} The reranked chunks are injected into a system prompt that instructs the LLM to answer based on the provided context, cite sources, and indicate when information is not available.
    \item \textbf{LLM Generation:} The prompt is sent to the LLM (Kimi K2.5 via Moonshot AI's OpenAI-compatible API). The response is streamed back to the client using server-sent events for a responsive user experience.
    \item \textbf{Session Persistence:} The message and response are stored in the \tech{chat\_messages} table, linked to the user's chat session for future reference.
\end{enumerate}

The pipeline includes several optimisation decisions driven by empirical testing. The 20-candidate retrieval with reranking to 5 was chosen because raw vector similarity alone produces false positives --- semantically adjacent but topically irrelevant chunks. The Cohere reranker provides a cross-attention score that considers the query--document pair holistically, significantly improving precision. The 0.75 cosine distance threshold and 0.15 relevance floor were tuned against a test set of university lecture queries to balance recall and noise. Markdown syntax is stripped before embedding (removing headers, emphasis markers, code fences, and HTML tags) because formatting tokens dilute the semantic signal in the vector space.

\subsection{Vector Storage Design}

The \tech{app.chunks} and \tech{app.embeddings} tables are central to the RAG system. Each chunk row contains the text, its source document ID, and user ownership. A corresponding row in \tech{app.embeddings} stores the 1024-dimensional vector embedding, linked via \tech{chunk\_id}. The table uses a pgvector HNSW index for approximate nearest-neighbour queries, providing O(log~n) lookup performance.

The choice of 1024 dimensions was driven by two constraints: pgvector's HNSW index has a 2000-dimension limit, and the initial self-hosted Ollama embeddings at 4096 dimensions also ran through a Cloudflare Tunnel with response timeouts. Migrating to Cohere's \tech{embed-multilingual-v3.0} at 1024 dimensions resolved both, enabling HNSW indexing and eliminating the tunnel bottleneck. This pivot is discussed further in Section~\ref{sec:pivots}.

Notable migrations include the initial schema creation~(\tech{001}), OAuth accounts~(\tech{002}), quiz session card IDs~(\tech{003}), vault job columns~(\tech{005}), Canvas import uniqueness constraints~(\tech{006}), and email verification support. This iterative approach reflects agile development, with the schema adapting to new requirements as features were built.

\section{CI/CD \& Deployment}

\subsection{AWS Amplify Pipeline}

The project uses AWS Amplify Gen~2 as the primary CI/CD platform. The build and deployment pipeline is triggered automatically on each push to the \tech{main} branch. The pipeline executes the following stages:

\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{Install:} Runs \tech{npm ci} to install dependencies from the lockfile.
    \item \textbf{Build:} Executes \tech{next build}, producing optimised static assets and serverless functions.
    \item \textbf{Deploy:} Distributes the build output to Amplify's CDN and deploys SSR routes as Lambda@Edge functions.
\end{enumerate}

Environment variables (database credentials, API keys, AWS service endpoints) are managed through Amplify's environment configuration, with separate variable sets for the \tech{dev} and \tech{main} branches.

\subsection{Branch Strategy}

The repository follows a two-branch model. The \tech{dev} branch serves as the integration branch where features are developed and tested. When stable, changes are merged to \tech{main} via pull request, triggering the production deployment. Both branches have corresponding Amplify environments with independent configurations.

\subsection{ECS Fargate Deployment}

The Canvas import worker runs as a separate ECS Fargate service. Its container image is built from a Dockerfile in the repository and pushed to Amazon ECR. The ECS task definition specifies resource limits, environment variables, and the SQS queue URL. The service is configured with a desired count of zero by default, scaling up when SQS messages are detected.

\section{Design Decisions \& Architectural Pivots}
\label{sec:pivots}

Over the course of development, the team made several significant architectural pivots. Each represents a real engineering decision driven by constraints encountered during implementation. These pivots are documented here as evidence of iterative design and practical problem-solving.

\subsection{Pivot 1: Notea Fork to Custom Next.js Application}

The project initially began with Notea, an open-source note-taking application, as a starting point. The intent was to extend it with RAG capabilities and Canvas integration. However, Notea's architecture did not support the vector search, LMS integration, or AWS infrastructure required by OghmaNotes. The team ultimately stripped Notea down to its core and rebuilt most components, migrating from the fork to a clean Next.js application.

\textbf{Lesson:} Forking an existing project can save time initially but creates technical debt when the original architecture diverges significantly from the target design. Starting fresh would have been faster in retrospect.

\subsection{Pivot 2: Monorepo to Single Application}

The project started as a monorepo with separate packages under an \tech{apps/web/} directory. This introduced complexity in the Amplify build configuration that proved disproportionate to the benefits for a three-person team. The team collapsed the structure into a single Next.js application, simplifying both development and deployment.

\subsection{Pivot 3: BullMQ (Redis) to AWS SQS}

The Canvas import job queue was initially implemented using BullMQ backed by ElastiCache (Valkey/Redis). This worked in local development but failed in production: Amplify's Lambda functions could not reach ElastiCache without VPC configuration, and the Amplify Lambda runtime had a 29-second timeout that was incompatible with long-running import jobs. The team migrated to AWS SQS, which uses IAM authentication (no VPC required) and decouples the API from the worker entirely.

\textbf{Lesson:} Cloud networking constraints drive architectural decisions more than theoretical best practices. SQS was the pragmatic choice given the deployment environment.

\subsection{Pivot 4: Self-Hosted Embeddings to Cohere}

The initial plan was to run embeddings on a self-hosted Ollama instance (qwen3-embedding, 4096 dimensions) and a reranker on the developer's homelab, exposed via a Cloudflare Tunnel for secure remote access. Two problems emerged: first, Cloudflare's tunnel enforces a response timeout that is incompatible with inference latency on heavier embedding models; second, the 4096-dimensional vectors exceeded pgvector's HNSW index limit of 2000~dimensions, forcing brute-force O(n) scanning for every query. Cohere's \tech{embed-multilingual-v3.0} API resolved both issues: managed API with no timeout constraints, and 1024-dimensional vectors compatible with HNSW indexing. The cost was low enough that the tradeoff was straightforward.

\textbf{Lesson:} Cloudflare Tunnel response timeouts are a hidden constraint when routing inference traffic through a homelab. Managed APIs eliminate the tunnel entirely.

\subsection{Pivot 5: OpenWebUI to Direct LLM API}

The RAG pipeline initially routed LLM requests through a self-hosted OpenWebUI instance, also behind the Cloudflare Tunnel. The same timeout constraints applied, and the additional hop added latency without benefit. The team migrated to direct API calls to Moonshot AI's Kimi K2.5, which provides an OpenAI-compatible endpoint at low cost, removing the tunnel dependency and the OpenWebUI authentication layer entirely.

\subsection{Pivot 6: pnpm to npm}

A mid-project package manager switch from pnpm to npm was necessitated by compatibility issues with the Amplify build environment. While pnpm offers faster installs and stricter dependency resolution, its workspace features conflicted with Amplify's build process. The switch to npm resolved build failures with minimal disruption.

\subsection{Budget Constraints}

The project operated within an AWS education credits budget. This constraint directly influenced infrastructure decisions: the self-scaling ECS Fargate pattern (scaling to zero when idle) was chosen specifically to minimise compute costs. Instance sizes were kept minimal (\tech{db.t3.micro} for RDS, \tech{cache.t3.micro} for ElastiCache), and services were deployed in a single region (\tech{eu-north-1}) to avoid cross-region transfer costs.

\subsection{Testing Patterns}

The test suite follows several consistent patterns. Module mocking uses \tech{vi.mock()} with \tech{vi.hoisted()} for dependency injection, allowing tests to run without live database or API connections. Environment variable testing uses \tech{vi.stubEnv()} with cleanup via \tech{vi.unstubAllEnvs()} in \tech{afterEach} blocks. Timer-dependent logic (debounce, rate limiting) uses \tech{vi.useFakeTimers()} with explicit advancement. Error paths are tested alongside success paths --- for example, the crypto tests verify that decryption with a wrong user ID throws, and that tampered ciphertext is detected.

\subsection{Honest Assessment}

The 39-file test suite provides solid coverage of backend utilities, validation, and security-critical paths (cryptography, authentication, input validation). However, coverage has clear gaps. Frontend React components have no automated tests --- all UI verification was manual. The Canvas import worker's end-to-end flow (SQS polling, Canvas API interaction, S3 upload, OCR, embedding) is tested at the unit level for individual functions but lacks an integration test that exercises the full pipeline. The RAG query flow similarly has unit tests for chunking, embedding, and reranking, but no test that simulates a complete query-to-response cycle.

End-to-end testing was performed manually during development, with each feature verified against requirements before merging. The team acknowledges that the BullMQ-to-SQS migration failure --- which only manifested in production due to VPC networking constraints --- would likely have been caught by an integration test that ran against a production-like environment. This experience reinforced the value of testing infrastructure interactions, not just application logic.

\section{Challenges \& Reflections}

\subsection{Semyon Fox}

As technical lead, my biggest challenge was scope management. The combination of AI-assisted development and genuine ambition created a feedback loop: every feature felt achievable, so we kept adding them. The project grew from a simple notes app to a full-stack platform with RAG, Canvas integration, spaced repetition, and a GPU-backed OCR pipeline --- all within a university project timeline.

The most technically demanding work was the AWS infrastructure. Debugging why Amplify's Lambda functions couldn't reach ElastiCache (VPC networking), why SQS messages weren't being consumed (IAM permissions), and why Marker's GPU instance took 15 minutes to cold-start (pip installing marker-pdf on every boot, solved by pre-baking a Docker image) each consumed entire days. These problems don't appear in tutorials or documentation --- they emerge from the specific intersection of services in your architecture.

Managing a three-person team while being the primary implementer created an inherent tension. I wanted to empower my teammates to own their contributions, but the interconnected nature of the system meant that integration work often fell to me. In retrospect, I should have invested more time in documentation and interface contracts early on, so that components could be developed more independently.

The key lesson from this project is that architecture decisions made under time pressure have long-lasting consequences. The BullMQ-to-SQS migration, the Ollama-to-Cohere embedding switch, and the monorepo collapse each cost days of rework. Building a proof-of-concept for each integration point before committing to it would have saved significant time.

\subsection{Samuel Regan}

For me, the most glaring challenge wasn't writing code. It was the learning curve and the invisible layer between the code and the services.\newline
In a project of this magnitude, there were so many components from the AWS services such as instances and environment variables that when something failed even getting the logs for the deployed project was a huge endeavour. Most of the time the code would fail silently and it was almost always not an internal server error, but a configuration issue. For example, I spent an entire day debugging why the rag pipeline worker queue was not dispatching ingestion requests, and when it did it couldn't find the database or persist any data. The root cause was a second database instance running in parallel. My IDE was pointed to a different, older one, so the SELECT checks always returned empty chunk columns.

The ambition for this project between myself and my teammates compounded this. This was our first chance to develop a real full stack app alongside our course, and thus we had chosen the technologies and methodologies we believed would be most suitable for the project and consequently, I was overwhelmed with a long list of technologies, such as TypeScript, Next.js, and a full array of AWS services, as well as learning team Git workflows for the first time which all formed into a very harsh learning curve.

Upon reflection, I have learnt that on a production level, programming skill and familiarities with the technologies are equally important. For the first half of this project I neglected the services making the production possible and focused on the backend code, and that tunnel vision backfired greatly when I tried to integrate my features into the rest of the system.
That said, thanks to this ambitious project I have become far more familiar with cybersecurity, the connection between frontend and backend functions/services such as features being too slow for a UI facing application,integrating LLM models and manipulating them for niche purposes, the services required to host an application on AWS, and the critical thinking required to invent my own solutions to problems, such as our RAG pipeline ingesting too slowly.
The very worker queue I described debugging earlier was my solution to that problem, processing documents in an ordered queue with workers rather than piling on large datasets to be ingested at once.


\subsection{Shreyansh Singh}

Working on OghmaNotes was my first experience with production cloud infrastructure. Setting up AWS RDS with SSL certificates, configuring S3 bucket policies, and getting SES verified for transactional email each presented their own learning curves. The most challenging aspect was understanding how the various AWS services connect --- security groups, IAM roles, and VPC configurations are not intuitive when coming from a local development background.

The OAuth implementation with Auth.js required careful attention to the callback flow and account linking logic, particularly handling edge cases where a user might register with email and later try to sign in with Google using the same email address. The Canvas API integration involved navigating a large REST API surface and understanding the Canvas data model (courses, modules, items, files), which required significant research and experimentation.

The key takeaway from this project was that real-world software engineering is as much about infrastructure and integration as it is about writing application code. The skills I developed in AWS, database design, and API integration are directly applicable to professional work.

\section{Future Work}

Several improvements and features are planned or under consideration for future development:

\textbf{Mobile Application:} The current application is responsive but not optimised for mobile workflows. A dedicated mobile application (or Progressive Web App) would better serve students who want to review notes on the go.

\textbf{Hybrid RAG Search:} Combining vector similarity with BM25 keyword search and supporting additional document formats would improve retrieval precision, particularly for technical terms and proper nouns.

\textbf{Collaborative Editing:} Real-time collaborative editing would allow study groups to work on shared notes simultaneously, integrated with the RAG system.

\textbf{Expanded LMS Support:} Adding integrations for Moodle, Blackboard, and other learning management systems would broaden the platform beyond Canvas.

\textbf{BYOK (Bring Your Own Key):} Allowing students to connect their own LLM API keys would provide model choice and reduce platform costs. The settings UI stub already exists.

\textbf{Offline Support (PWA):} A Progressive Web App with IndexedDB caching would allow offline note review and flashcard study, syncing when reconnected.
